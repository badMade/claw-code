## 2024-04-09 - Rust String Cloning Optimization
**Learning:** In Rust, building lists of string parts for joining (e.g., `vec![string1.clone(), string2.clone()].join(" ")`) is a common pattern that can lead to unnecessary heap allocations. This codebase frequently does this when formatting reports. When the source values are already `String`s, borrowing them as `&str` and pushing them to a `Vec<&str>` before calling `.join()` entirely avoids these intermediate heap allocations. Additionally, calling `.to_string()` on static string slices just to appease a `Vec<String>` is wasteful when `Vec<&str>` works perfectly.
**Action:** When constructing strings from parts using `Vec` and `.join()`, look for opportunities to use a `Vec<&str>` populated with borrowed string slices (`.as_str()`) instead of a `Vec<String>` populated with cloned strings (`.clone()`).
## 2024-04-24 - Python Cached Dictionary Lookup Optimization
**Learning:** In Python loops performing O(N) linear searches where a target string is repeatedly matched against the name properties of objects in a sequence (e.g., `for module in PORTED_TOOLS: if module.name.lower() == needle: return module`), the repeated calls to `.lower()` string methods within the loop and the O(N) scanning time are inefficient. Replacing the linear search with a lazily initialized and cached O(1) dictionary lookup (using `@functools.lru_cache(maxsize=1)` and returning `types.MappingProxyType` to prevent unintended modifications) reduces time per call dramatically (e.g., from ~7.1 microseconds down to ~0.3 microseconds, a 21x improvement). The lookup preserves exact search semantics like first-match priority for duplicate keys by adding items to the cache only if they don't already exist (`if name_lower not in lookup: lookup[name_lower] = module`).
**Action:** When implementing repetitive lookup functions returning objects from static lists (like `get_tool` and `get_command`), use a module-level dictionary or an `lru_cache(maxsize=1)` dict mapping lowercased names to the objects to provide O(1) lookups instead of O(N) loops.
