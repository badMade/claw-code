## 2024-04-09 - Rust String Cloning Optimization
**Learning:** In Rust, building lists of string parts for joining (e.g., `vec![string1.clone(), string2.clone()].join(" ")`) is a common pattern that can lead to unnecessary heap allocations. This codebase frequently does this when formatting reports. When the source values are already `String`s, borrowing them as `&str` and pushing them to a `Vec<&str>` before calling `.join()` entirely avoids these intermediate heap allocations. Additionally, calling `.to_string()` on static string slices just to appease a `Vec<String>` is wasteful when `Vec<&str>` works perfectly.
**Action:** When constructing strings from parts using `Vec` and `.join()`, look for opportunities to use a `Vec<&str>` populated with borrowed string slices (`.as_str()`) instead of a `Vec<String>` populated with cloned strings (`.clone()`).

## 2024-05-18 - execution_registry O(1) Lookup
**Performance Bottleneck:** Lookups via `ExecutionRegistry.command` and `ExecutionRegistry.tool` were previously iterating sequentially through the lists. Due to the case-insensitivity of the check, `command.name.lower()` string allocation was happening $O(N)$ times on *every* call.
**Learning:** `functools.cached_property` on immutable `@dataclass(frozen=True)` containers allows lazy map initialization. When a duplicate name collision handling is "first match wins" (like with JSON tool snapshots), use `if lowered not in lookup: lookup[lowered] = command` during initialization. This changes the O(N) linear sweep with strings to a singular O(1) access and string allocation. Python 3.12+ makes using cached properties on frozen dataclasses natively work without raising `FrozenInstanceError`.
